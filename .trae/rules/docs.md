# 项目文档管理规则

## 文档更新时机
当以下情况发生时，必须更新项目文档：
- 新增子项目
- 删除子项目
- 子项目结构发生重大变化
- 子项目技术栈变更
- 子项目git提交仓库更新开发日志内容

## 文档位置和命名

- **文档存储位置**：`d:\@code\2026\AwardWebsites\docs` 文件夹
- **文档命名规则**：`<子项目文件夹名称>.md`
- **文档格式**：Markdown 格式

## 项目列表参考

项目根目录下的子项目列表参考 `d:\@code\2026\AwardWebsites\README.md` 文件中 `Websites` 部分的内容。

## 文档创建和更新流程

### 新增子项目时
1. 在项目根目录创建新的子项目文件夹
2. 在 `README.md` 的 `Websites` 部分添加项目信息
3. 在 `docs` 文件夹中创建对应的 `.md` 文档
4. 文档应包含以下内容：
   - 项目简介
   - 技术栈
   - 项目结构
   - 安装和运行
   - 开发日志

### 删除子项目时

1. 删除子项目文件夹
2. 从 `README.md` 的 `Websites` 部分移除项目信息
3. 删除 `docs` 文件夹中对应的 `.md` 文档

### 更新子项目时

1. 修改子项目代码
2. 更新 `docs` 文件夹中对应的 `.md` 文档，反映最新的项目结构和功能

## 文档模板

每个子项目文档应遵循以下模板结构：

```markdown
# <项目名称>

## 项目简介
简要描述项目的目的和功能

## 技术栈
分点展示技术栈，包括框架、构建工具、UI 库等

## 项目结构
树状展示文件夹结构，文件夹功能
## 安装和运行
包含安装依赖、开发模式和构建生产版本，node版本要求说明
## 开发日志
记录项目开发过程中的重要事件、问题和解决方案（可以在代码提交之后进行记录更新），按照日期从最近到最远的顺序排列。
例如
### 2025-12-12
- 新增子项目 follow.art
```



## 注意事项
- 所有文档必须使用 Markdown 格式
- 文档内容应保持最新，与实际项目结构一致
- 文档应清晰易懂，便于其他开发者理解项目
- 更新文档，保持原有内容的基础上，新增或修改必要的部分
